#!/usr/bin/env -S bun ./pad.shebang.tsx pad-diff.diff.pad.htm pad-diff.diff.pad.htm Preserve simple PAD A `.pad.htm` file remains executable HTML that opens in the browser and autosaves edits back to itself. PAD is only plain prose with no durable semantic structure for agents or control loops. PAD becomes an app framework with routing, storage, schema attributes, build steps, and admin UI. Executable HTML stays simple while custom elements add enough semantic structure for humans, agents, and generated snapshots. The current repo already satisfies the modest PAD spec: executable file, local server, editable `pad-text`, autosave, hot reload, and tests. The implementation is slightly too thin for diff-pad use because semantic work state exists only as prose. Register inert diff-pad elements and let the browser assign missing opaque IDs so humans never invent handles. Story-local control loop The primary organization is by story, not by global SPEC, REAL, GAPS, and NEXT sections. Wishes, observations, files, gaps, and work for one story are scattered across unrelated global sections. Everything is crammed into one undifferentiated story body with no lifecycle facets. Each story contains local `story-spec`, `story-real`, `story-gaps`, and `story-next` facets. The old PAD runtime had only `pad-document`, `pad-text`, and `pad-status`. That was healthy for the first spec but too thin for story-local change control. The new grammar should stay inert and structural before any regeneration, cron, or agent dispatch machinery is built. The design needs to move away from scattered global sections toward story-local lifecycle facets. The concept risks adding too many element names before the smallest useful loop is proven. Add this self-applying `.diff.pad.htm` as a tactile example before building automation. Style story facets so SPEC, REAL, GAPS, and NEXT are visually scannable inside each story. Do not build cron regeneration until this static grammar feels good as a hand-written control surface. Legible measured specs Boolean specs, golf specs, and needle-thread specs can coexist inside one story. Specs are only prose, so the loop cannot show numeric progress or directional overshoot. Specs become embeddings or scores that are impossible for the human to audit quickly. Specs define named axes in prose, and gap snapshots expose legible vectors with distance, target, and confidence. The project has no generated scoring yet, but the proposed HTML can already carry named vector measurements. The hand-written example can show the vector shape, but actual regeneration is still future work. Later, teach tooling to regenerate `gap-vector` from `spec-razor`, `spec-thread`, `spec-golf`, and observed reality. Healthy perpetual motion Specs and gaps go stale; no pressure is regenerated. Automation creates endless churn and agents work without human-shaped intent. Cron refreshes observations, gaps, and proposed work while approval gates execution. Some growth gaps should intentionally remain open or be replaced by higher-order gaps so the project keeps producing desired work patterns. There is currently no cron loop, no generated gap snapshot, and no approval workflow. The project leans stale/manual today, but jumping straight to autonomous execution would overshoot into treadmill territory. Define what future regeneration may overwrite: generated observations, gaps, and proposed work, but not human-authored SPEC or approved work. OpenCode SDK orchestration All future AI agent and LLM orchestration for diff-pad regeneration uses `@opencode-ai/sdk` instead of direct provider APIs or ad hoc shell prompts. Regeneration scripts call model providers directly, parse loose prose, and bypass the editor/session context that OpenCode already owns. PAD grows its own agent platform with duplicated sessions, provider config, event streams, file search, and permission semantics. Diff-pad automation treats OpenCode as the control plane: create/connect a client, use sessions for prompts, request structured output for generated snapshots, subscribe to events, list agents, and inspect files through SDK APIs. The OpenCode SDK docs expose the primitives needed for this loop: `createOpencode`, `createOpencodeClient`, `app.agents`, `session.create`, `session.prompt`, structured JSON schema output, `event.subscribe`, `find.*`, `file.read`, and `file.status`. This repo should not add `@opencode-ai/sdk` as a dependency until there is an executable regeneration or dispatch script that imports it. The static grammar now names generated gaps and proposed work, but it does not yet define the SDK-backed regeneration contract. The likely next risk is chaos: generated outputs need focused JSON schemas so agents update the right story facets without trampling human-authored SPEC or approved work. Define structured-output schemas for regenerated `story-real`, `story-gaps`, and proposed `work-item` snapshots. Prototype a Bun script that uses `createOpencode` or `createOpencodeClient`, sends one story to `session.prompt`, and receives a typed gap/work JSON payload. Use `app.agents` and `event.subscribe` only after approved work dispatch exists; keep cron regeneration observational until then.
